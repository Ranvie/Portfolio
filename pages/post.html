Mais alguns detalhes sobre o Projeto de Software

Autor: Rangel

Estrutura do Software

Existem muitos conceitos (muitos mesmo) no projeto de software, e eles servem exatamente para nos passar a experiência de outras pessoas que já tiveram diversos problemas quanto a gestão de um programa, aplicativo, etc. Neste caso, deixaram formas não apenas para os contornar, mas sim, evitar que eles aconteçam.

Existem diversos passos, procedimentos e estruturas criadas, que visam auxiliar na documentação e na codificação de um programa. Com isto, segue-se:

Abstração

Representar os procedimentos, dados e controle em vários níveis de detalhes. Se define uma solução usando linguagem de ambiente do problema.

Refinamento

melhoramentos sucessivos, trazendo níveis de detalhes cada vez melhores. Método top down (ver todos os detalhes genéricos e depois ir para os individuais) e bottom up (ver os detalhes individuais antes e depois ir abrangindo para algo geral).

Modularidade

Quebrar um problema maior em pequenos módulos gerenciáveis, afim de diminuir o grau de complexidade unitário. Estes tem diferentes níveis de independência funcional, e possuem como característica a coesão (se módulo faz apenas aquilo que foi desenvolvido para fazer) e o acoplamento (a função é chamada em outra quando necessária para fazer um cálculo, tarefa e etc (exemplo do foguete))

Arquitetura

Agrupa módulos que solucionam um problema do mundo real em algo mais genérico.

Hierarquia de Controle

Representa graficamente as chamadas de procedimentos dentro de um software (é como se mostrasse uma chamada que mostra outra, retorna para outra, etc, em formato de árvore)

Estrutura de dados

Representa vetores, matrizes, árvores, grafos e etc.

Procedimento de Software

Detalha o processamento de cada módulo individualmente, definindo os eventos, pontos de decisão e diversos outros conceitos relacionados a como cada procedimento/módulo age enquanto está exercendo sua função.

Ocultação de Informações

O que ficará visível das informações manipuladas em um método, exemplo do public, private e protected em orientação a objetos ou escopo em linguagens procedurais.

Métricas para definir custo e tempo de desenvolvimento

Este é um tópico amplamente abordado e que muitas pessoas não sabem nem por onde começar. Clientes adoram softwares baratos e que sejam rápidos para ficarem prontos, então é de responsabilidade da organização de parametrizar os requisítos e dizer o custo e o tempo que levará para ficar pronto. É muito importante fazer essa etapa, não apenas para satisfazer o mais exigente dos clientes, mas também para ter noção da viabilidade de um projeto, afinal, de nada adianta investir R$10.000 em um projeto que retornará R$2.000.

Existem diversas técnicas para definir esses valores, por exemplo, a utilização da contagem de linhas de código, onde, para cada certa quantidade de linhas, é considerado um tempo e um custo, entretanto, esse modelo serve única e exclusivamente para a linguagem da qual foi desenvolvido, então, para cada uma deve ser feito um estudo ao definir os valores. Felizmente, existem outros métodos e aqui estarei abordando o método de pontos de função.

Por mais que seja mais complexo, se torna abrangente e não depende de linguagem. A métrica orientada a função se baseia em pontuar por níveis de complexidade um software (simples, médio, complexo), levando em consideração aspectos mais técnicos como a quantidade dados, arquivos de erro, chaves, tabelas, telas e outros objetos, que ajudam a quantificar uma pontuação de custo e tempo estimados do software.

Aula sobre Projeto de Software

Autor: Rangel

Qual a importância do Projeto de Software?

Essa é uma etapa pré implementação. Nela são definidos os modelos que vão fazer parte do software final, definindo coisas como arquitetura, design e layout, algoritmos e outros. É importante fazer esse processo pois, além de poupar recursos financeiros, tempo e atribuir uma organização interessante aos projetos, evita que existam inconformidades e despadronizações no mesmo. Imagine que diversas equipes costumam trabalhar em paralelo em um mesmo projeto, sem essa abordagem, cada equipe ou até mesmo integrante, pode querer atribuir suas próprias interfaces, nomes de variáveis, tabelas em bancos de dados desconexas com os demais e diversos outros problemas que com certeza afetarão a qualidade final do produto.

Dramatização da importância do Projeto de Software. Disponível em: <https://docs.google.com/presentation/d/1_4DEr6vwFU6redoT8yYc2jrQDgFk-fWB/edit#slide=id.p6>. Acesso em: 30 abr. 2022.

Os tipos de projeto

Existem diversos tópicos a serem abordados quanto a criação de um projeto de software, e são divididos em duas categorias. Temos o ponto de vista gerencial (Projeto Preliminar, Projeto Detalhado) e o ponto de vista técnico, do qual, vou abordar alguns de seus tópicos nesta postagem.

  1. Projeto de Dados
  2. Projeto de Interface

Projeto de Dados

Nessa etapa a parte das informações do projeto são organizadas. Se verifica quais são os tipo de dados trabalhados assim como as estruturas também. Nesta parte, é feito um dicionário de dados, que serve como uma documentação especificando quais os tipos de variáveis usados no projeto, bem como suas funções, seu tipo, o que pode ou não receber e outros metadados atrelados a estes tipos. Existe também o dicionário para estrutura de dados, que especifica tipos que não são definidos por padrão, sendo assim, o desenvolvedor deve os criar. Um exemplo é um campo de um banco de dados que recebe itens como PDFs, GIFs, JPGs e outros, que não possuem uma implementação prévia por parte daqueles responsáveis pela criação de uma linguagem quaisquer.

Também é nesta etapa que são definidos os padrões a serem seguidos quanto a utilização de funções e variáveis dentro de um sistema. Como citado anteriormente, as chances de fracasso em um projeto são notavelmente maiores quando não existe uma padronização decente, pois cada um seguirá seus próprios padrões, o que trará incongruências, problemas de utilização dos recursos já criados, muitas vezes duplicidade de funções e diversos outros problemas.

Projeto de Interface

Este é um aspecto muito importante, ele que define muitas vezes se um projeto dará certo ou não, por parte do público. Uma coisa comprovada, é que a primeira coisa que as pessoas reparam ao utilizar um produto, é sua interface. Portanto, interfaces não atrativas afastam o público e consequentemente reduzem as chances de sucesso de um projeto.

Outro ponto é a facilidade e acessibilidade das funções em um software, de nada adianta uma interface bonita e muito confusa, da qual trás mais problemas do que soluções. Neste caso entram aspectos de design em UI e UX e que por muitas vezes são uma área completamente difundida na criação de produtos.

É importante cuidar muito bem dos aspectos gráficos em uma interface, as cores são um desses, já que muitas pesquisas demonstram que estas podem ser utilizadas para passar sensações, por exemplo, a cor preta passa um efeito de modernismo, enquanto a cor vermelha trás força, paixão, raiva e outros. Essa área de estudo se chama Psicologia das Cores.

Aula sobre Requisitos e Modelo de Casos de Uso

Autor: Rangel

Levantamento de requisitos, o que é?

Esse é um processo de suma importância feito sempre na primeira etapa de todo projeto, nele são debatidos junto a parte interessada, o que deve conter em determinada aplicação. Tais informações devem ser muito bem especificadas e são documentadas para serem utilizadas posteriormente. Dependendo do tipo de metodologia aplicada ao projeto, essa fase ocorre uma ou mais vezes até a entrega final do produto.

Para melhor compreender os requisitos, existem alguns tipos que são gerados enquanto nas fases de planejamento, e eles podem ser agrupados em:

  1. Funcionais
  2. Não funcionais
  3. Usuário
  4. Sistema

Requisitos de Usuário

São aqueles ditos pela parte interessada, da qual muitas vezes não tem conhecimento técnico, mas precisa de determinadas funções em um produto. Por exemplo, uma página que permite a criação e login de usuários, fazer postagens, comentários, mostrar notificações a aqueles que acessarem e outras funcionalidades. São genéricos e não especificam como deve ser feito, apenas o objetivo da funcionalidade.

Requisitos de Sistema

Já são feitos pelo time de desenvolvimento, que especificam de forma abstrata como determinada funcionalidade deve ser implementada, utilizando de determinadas ferramentas, linguagens e afins. Envolve a modelação de como o sistema deverá funcionar, com quais dados ele contará, os tipos dos dados e outras informações pertinentes para a implementação.

Requisitos Funcionais

São aqueles que alteram diretamente o funcionamento de um sistema e definem exatamente quais tarefas e funções que o mesmo terá, de forma específica e abstrata para cada item dos requisitos de sistema.

Requisitos Não Funcionais

São importantes, mas não implementam nada ao sistema diretamente, como por exemplo, verificações legais, os stackholders pedirem uma interface bonita e intuitiva, a utilização de determinados padrões impostos pelo cliente e diversos outros exemplos.

Exemplos de requisitos não funcionais. Disponível em: <https://docs.google.com/presentation/d/1Oxc47eupagmMn6kNpWrObO3RBi_aiucu/edit#slide=id.p4>. Acesso em: 29 abr. 2022.

Tipos de requisitos com FURPS+

Como já deve ter percebido, os requisitos tratam de diversas categorias diferentes e sem uma organização fica difícil de dizer do que se trata e qual sua contribuição ao projeto. Por conta disto, o modelo FURPS+ foi criado. Seu propósito é categorizar requisitos de modo a facilitar a compreensão dos mesmos e dizer qual utilidade terá dentro de um projeto.

Existem os seguintes tipos:

  1. Funcionalidade (Functionality)
  2. Usabilidade (Usability)
  3. Confiabilidade (Reliability)
  4. Desempenho (Performance)
  5. Suportabilidade (Supportability)

Funcionalidade

Característica definida, capacidades, a generalidade, a segurança.

Usabilidade

Fatores humanos, estética, coerência, documentação.

Confiabilidade

Gravidade de frequência de falha, de capacidade de recuperação, a previsibilidade, a rigor, tempo médio para falha.

Desempenho

Velocidade, eficiência, tempo de resposta de consumo, taxa de transferência de recursos.

Suportabilidade

Testabilidade, extensabilidade, adaptatibilidade, facilidade de manutenção, compatibilidade, configurabilidade, facilidade de manutenção, instalabilidade, localizabilidade, portabilidade.

O termo "+" no nome FURPS se refere a outros aspectos como implementação, interface, operações e questões legais.

Casos de uso

Essa é uma ferramenta, textual, muito interessante, com ela é possível definir atores e ações para completar determinadas tarefas em um sistema, como implementação de uma nova função. Por mais seja geralmente encontrado em versão gráfica, os autores dizem que este é um modelo textual de documentos, não diagramático, portanto vou estar tratando da versão em tabelas. Uma prática muito comum, chamada caso de uso caixa preta, é descrever de forma como o usuário descreveria uma determinada função. Por exemplo, ao dizer que um botão deve registrar um novo usuário, não se deve dizer como, nem o processo para isto acontecer, apenas, o que o botão faz, e neste caso, registrar uma nova conta. Existe também o caso de uso caixa branca, que é uma maneira completa de se descrever como um processo deve ocorrer para concluir aquela ação.

Exemplo de um caso de uso gráfico (por mais que a ferramenta seja textual). Disponível em: <https://pt.wikipedia.org/wiki/Diagrama_de_caso_de_uso>. Acesso em: 29 abr. 2022

A sua formalidade pode ser separada em:

  1. Resumido: um parágrafo (caso de sucesso)
  2. Informal: múltiplos parágrafos (vários cenários)
  3. Completo: descrição detalhada de condições e garantias de sucesso

Neste modelo existem os fluxos e desvios. Isso significa que a modelação já presupõe eventos que podem levar a resultados diferentes. Por exemplo, caso o usuário não digite um valor válido em uma caixa que apenas aceite dígitos.

Palestra na matéria de Engenharia de Software: Ferramentas de Pesquisa

Autor: Rangel

Introdução

O objetivo da palestra foi de ensinar meios de filtrar o que se pesquisa, a fim de encontrar melhores resultados para as buscas, sendo elas em catálogos, motores de busca e outros.

Foi voltada para as ferramentas disponíveis aos alunos do instituto, e dentre elas as ferrementas apresentadas foram:

  1. Catálogo Pergamum
  2. Bibilioteca Pearson
  3. Portal de Periódicos da CAPES

Catálogo Pergamum

Essa ferramenta permite pedir empréstimos do acervo bibliotecário, não somente do campus Birigui, como de qualquer outra biblioteca de outro campus que esteja registrada. Caso não haja o documento ou livro no campus do aluno, mas esse documento ter disponibilidade em outro, é possível fazer uma requisição à bibilioteca que pedirá esse livro ou documento emprestado para o requerente.

Dentre outras funcionalidades como:

  1. Visualizar histórico de livros
  2. Ver as reservas
  3. Renovar livros (Até 3 vezes)
  4. Consultar disponibilidade de acervo em outros campus
  5. Requerimento de nada consta (CUIDADO!!!, ao clicar nesta opção seu usuário é encerrado e não poderá mais acessar os acervos da biblioteca)
  6. Alertas de últimos documentos devolvidos à biblioteca
  7. Emissão da Ficha Catalográfica

Para acessar essa ferramenta, entre no site do catálogo pergamum e coloque os dados de login ou entre diretamente pelo SUAP de seu campus.

Biblioteca Pearson

Possui um grande acervo literário, online, disponível para os alunos do IFSP. Conta com diversos livros de várias áreas de conhecimento diferentes, por exemplo, engenharia, matemática, química, física, e outras.

Algumas de suas funcionalidades são:

  1. Continue lendo
  2. Listas de leitura
  3. Pesquisa pelo título de um livro, categoria, sub-categoria e outros
  4. Impressão de páginas utilizando créditos comprados dentro da plataforma
  5. Marcar páginas
  6. Marcações e notas
  7. Cartões de estudo gerados pelo usuário

O acesso a ferramenta se dá pelo site da bibilioteca Pearson ou por um link contido na página inicial do SUAP do campus do aluno.

Portal de Periódicos da CAPES

Esse é um site muito interessante, seu objetivo é trazer periódicos (revistas, artigos e outros) técnicos, científicos e outros, de grande nível de confiabilidade pois são revisados por uma equipe de profissionais da área. A utilização deste material é indescritivelmente importante quanto a validação das informações usadas ao se fazer um projeto (seja um artigo, estudo, TCC e outros), visto que muitas instituições pedem informações de confiança ao verificar um trabalho, e costumam classificar estes em níveis. (A, B, C, ...)

Para acessar esta ferramenta existem duas maneiras:

  1. Enquanto fisicamente no IFSP, acessar o site fará com que ele reconheça sua posição e faça login automaticamente
  2. Se estiver fora, ainda é possível acessar, mas terá que fazer login manualmente

Em caso de dúvidas existe este tutorial feito pelo IFSP.

Meu primeiro dia na matéria de Engenharia de Software

Autor: Rangel

UML e PU

O primeiro dia foi muito interessante (e um pouco desesperador), uma vez que não tenha feito as disciplinas anteriores sobre gestão de software, conheci várias ferramentas (algumas já havia ouvido falar como metodologias em cascata e SCRUM), que são muito importantes na hora de se levantar requisítos e organizar como o desenvolvimento se dará.

UML é um exemplo de ferramenta que não conhecia mas que traz uma forma de organização gráfica muito interessante. Já havia ouvido falar por meio de um professor do instituto mas não havia procurado sobre ainda, e olhando agora, parece como um fluxograma de tarefas.

Imagem ilustrativa sobre o modelo UML

Também foi citado o Processo Unificado, que é basicamente um meio de padronizar a criação de software e evitar problemas quando se incluir novos funcionários em uma empresa. Visa a não produção em escala, ou seja, os produtos são feitos sob medida para cada cliente. É um processo que passa por quatro etapas:

  1. Especificação do software (Pesquisa, levantamento de requisítos)
  2. Implementação (Operacional)
  3. Validação (Revisão)
  4. Evolução (Updates)

Há algumas práticas integradas a este modelo, e pode-se citar as seguintes:

  1. Desenvolvimento iterativo
  2. Gerenciamento de exigências
  3. Arquiteturas baseadas em componente
  4. Modelo visual
  5. Verificação intermitente de qualidade
  6. Controle de mudanças
Imagem ilustrativa sobre o Processo Unificado

Nessa metodologia especificamente, a forma de se utilizar é simples, há uma gráfico com quatro fases no topo e várias tarefas (disciplinas) em sua esquerda. Cada tarefa terá uma quantidade de tempo utilizado em cada fase, onde para o representar, se utiliza de formas onduladas, geralmente, que quanto maior sua área, mais tempo investido em determinada fase.

Engenharia Reversa - 27/05/2022

Autor: Rangel

O que é a engenharia reversa?

Esse é um processo que visa descobrir como um software funciona, e se faz bastante útil principalmente quando não se há nada concreto sobre o mesmo. A engenharia reversa geralmente acontece quando não se tem nada sobre o sistema, sejam códigos fonte, estruturas de dados e etc. Neste caso, a função do desenvolvedor será tentar entender o que está sendo feito, seguindo um caminho oposto ao da engenharia progressiva

Abstração

Esse termo se refere a ignorar informações que não são importantes para um projeto e dar ênfase naquelas que são relevantes. Isto é útil em um ambiente no qual não se tem conhecimento de um sistema, por exemplo, e este possui muitas funções. Neste ponto, é necessário tentar entender a intenção do desenvolvedor.

Existem dois termos mais utilizados neste caso:

  • Nível de abstração

    Quanto mais informações de um produto, menor o grau de abstração, o que torna se torna mais específico

  • Grau de abstração

    O quão específica é a informação

Assim como a engenharia progressiva, a reversa possui alguns passos definidos, que são:

  1. Implementacional

    Ideias de como funciona

  2. Estrutural

    Detalhes da linguagem de programação

  3. Funcional

    Funções e operações que o software faz

  4. Domínio

    Total conhecimento sobre como funciona, então constrói com o que foi aprendido

Propósitos da engenharia reversa:

  • Manutenção
  • Reuso

    Partes do código podem ser reutilizadas em outros locais, portanto, as seguintes atividades são feitas a fim de verificar se pode ou não ser reusado:

    1. Reconhecimento
    2. Decomposição
    3. Classificação
    4. Seleção
    5. Adaptação
    6. Composição
  • Garantia da Qualidade de Software

    Está em conformidade com os requisitos funcionais

O que é a ReEngenharia?

Por exemplo, vamos dizer que alguém foi contratado para fazer a manutenção em um sistema, do qual não possui documentação alguma. A primeira coisa que terá de fazer é entender como ele funciona, indo de informações de um grau de abstração menor para um maior, uma vez que os dados já estão sendo usados, mas terá de descobrir qual a função deles no sistema, como se estivesse levantando requisitos.

Redocumentação

Esse processo acontece em sistemas que não possuem documentos, no caso, requisitos, projetos de interface, etc. Neste caso, o desenvolvedor pega um sistema já pronto e começa a tentar ter um visão mais geral, saindo dos dados e chegando aos requisitos.

Para auxiliar este momento, existem algumas técnicas:

  1. Visualização do código
    • Program Slicing
    • Análise de dependência
  2. Entendimento do programa
    • Conhecimento do domínio das informações externas
    • Pattern Matching

O que é Reuso de software?

Sistemas já prontos, podem possuir componentes reutilizáveis, que podem ser integrados a outros softwares e trazer além de velocidade ao andamento do projeto, mais segurança, uma vez que estes elementos já foram previamente testados (ou espera-se), não tendo que se preocupar com a criação destes.

A engenharia reversa infringe a lei da propriedade intelectual?

Isso depende do contexto e do proprietário. Em casos onde o dono do software é a própria empresa, obviamente não há problemas, agora, em casos onde outra organização possui os direitos, esta tem total detenção do mesmo, mesmo que o código tenha sido escrito por seus funcionários. Neste caso, cópias para estudo são legais, entretanto, sua utilização indevida (distribuição, clonagem, afins) é proibida.

Gerência de Configuração de Software (GCS) - 19/05/2022

Autor: Rangel

O que é feito durante esta etapa?

Este momento no projeto é caracterizado pelas alterações que ocorrem durante toda a vida útil de um software, das quais, podem ser por exemplo, a estrutura dos dados, alterações no código fonte, mudanças nas ferramentas usadas e diversos outros.

Isto ocorre pois, um software não fica "parado no tempo", sempre existem modificações a serem feitas, das quais, sempre afetam a estrutura de dados e o código fonte, afinal, ao alterar um o outro terá efeitos colaterais e consequentemente precisará ser alterado também.

Existem vários motivos para se alterar um software, mas, os mais comuns são:

  1. Novas condições de negócio
  2. Novas necessidades
  3. Reorganização e/ou crescimento
  4. Restrições de orçamento ou cronograma

Portanto, é necessário ter o controle do que se foi alterado, tendo controle de versões e modificações, garantir que estas sejam aplicadas ao projeto e relatando o que foi feito.

Imagine que existem diversos artefatos gerados durante a criação de um software (especificações, planos de projeto, arquivos de código fonte, casos e planos de testes, manuais, arquivos de dados e muitos outros), dos quais se referem a uma versão específica do projeto (conhecida como baseline, vou entrar em detalhes mais a frente). É necessário saber o que se refere ao que, uma vez que, um problema no código fonte implica ter que voltar um backup, por exemplo. Nada garante que funcionará, pois, a estrutura do banco de dados poderá ter mudado, gerando incompatibilidades e causando mais problemas. Por isso, a GCS nos diz que devemos:

  1. Identificar os itens de configuração
  2. Gerir modificações em um ou mais destes itens
  3. Facilitar a construção de diferentes versões do mesmo produto
  4. Garantir a qualidade do software a medida que a configuração é feita ao longo do tempo

Itens de Configuração de Software (ICS)

Como citado anteriormente, é necessário saber quem são esses itens. Mas afinal, o que eles são? Simples, são a menor porção de dados no GCS. Em outras palavras, cada item individual, como um arquivo de código fonte, modelos de uso, modelos entidade-relacionamento e diversos outros. Que são gerados durante todo o processo da engenharia de software, no caso dos artefatos, porém é completamente válido se considerar as ferramentas usadas no desenvolvimento (navegadores, editores de texto, IDEs, compiladores e afins), que podem mudar a qualquer momento trazendo incompatibilidades ou funções que os devs desconhecem, trazendo vários problemas também.

Neste caso, cada item deve ter identificadores próprios, contendo nome, descrição, lista de recursos, data e outros, que sirvam como metadados para aquele determinado item. Isto facilita na catalogação de cada um e trás o benefício de, em eventuais erros do sistema, a consulta destes. Não ficando limitado ao código fonte e banco de dados.

Quando catalogamos esses itens e fazemos uma espécie de versão, criamos um ponto de retorno, do qual possui algo como uma snapshot do projeto, que pode ser acessada em casos onde é necessária uma ação rápida para não afetar os negócios dos clientes, por exemplo. Este ponto é chamado de baseline, ou linha de referência e como dito anteriomente, indica um estado completo de uma versão do software, contendo todos os artefatos exatamente como o software foi implementado.

Assim como todo outro artefato, os ICS podem (e devem, na medida do possível) ser:

  1. Identificados
  2. Analisados
  3. Corrigidos
  4. Aprovados
  5. Armazenados como uma configuração de software

Repositórios

Este é um conceito utilizado no versionamento, geralmente atrelado aos códigos fontes. Um exemplo pode ser o git que permite fazer alterações em um projeto, mantendo diferentes versões que podem ser acessadas em casos de falha em uma atualização, das quais criam uma imagem do projeto naquele momento. Elas possuem dados como: o que foi alterado, quem alterou, a data da alteração e diversos outros. Os repositórios podem estar localmente (git) ou remotamente (github).

Alguns termos são usados quanto aos repositórios, podendo citar o lock, que é um estado onde apenas um usuário pode alterar o código fonte por vez, check-out (pull, no git) que baixa as últimas alterações feitas em um repositório e check-in (push, no git) que envia alterações e/ou inserções ao repositório remoto.

Análise de Riscos - 12/05/2022

Autor: Rangel

Os riscos em um projeto

Essa etapa busca levantar os possíveis problemas que podem ocorrer durante a vida de um projeto. Por exemplo, perder alguém da equipe, eventuais percas de dados e equipamentos, mudanças no mercado que podem fazer o cliente querer outras funcionalidades, dentre diversos outros.

É importante fazer esse levantamento pois, quando já temos em mente o que pode ocorrer, podemos criar possíveis soluções, que evitam de termos que lidar com o fator surpresa, afinal, temos uma data estipulada para a entrega de um produto e atrasos implicam em custos e insatisfação do cliente.

Podemos fazer estimativas de risco a partir de:

  1. Experiência
  2. Acesso à informações históricas
  3. Dados qualitativos (não há medição ou métricas, apenas eventos)

Vale ressaltar que nenhum deles é exato por serem influenciadas por variáveis humanas, externas e afins.

Problemas que surgem ao fazer estimativas

Ao utilizar estes valores, estamos correndo o risco de cometer erros, mas ainda sim, é melhor que tentar dizer valores aleatórios. É necessário ficar atento a alguns detalhes quanto a essas estimativas, dos quais se seguem:

  1. O grau de incerteza das medidas
  2. Escopo mal compreendido
  3. Interessados que mudam o escopo do projeto

Grande parte dos problemas está contido nos dados e a sua qualidade, afinal, quanto menor a precisão deles, menos confiáveis são, mas como dito anteriormente, os problemas não são apenas internos (equipe, projeto), a parte interessada (clientes geralmente), apresenta grandes riscos quanto à estabilidade do produto, visto que podem começar a exigir mudanças que não são compatíveis com o projeto, e isso gera problemas complexos de serem resolvidos, o que consome tempo, recursos, etc.

Os fatores que aumentam os riscos em um projeto são a complexidade, o tamanho e o grau de estruturação. A complexidade vai depender da abstração feita no projeto, pois, ao se quebrar um problema em frações menores, se reduz a complexidade do mesmo, e fica mais simples de resolver. O tamanho, afinal, quanto maior, mais mão de obra, tempo e recursos é necessária, o que abre brechas para riscos internos e externos. E o grau de estruturação, que tem bastante a ver com tudo que a Engenharia de Software fala, um projeto sem um "norte" é como um navio que fica à deriva e pode vir a encalhar em algum momento.

Para diminuir os riscos são necessários dados históricos, que dizem de forma precisa, como um processo foi feito, quanto tempo levou, a mão de obra utilizada e fatores do tipo, que descrevem o esforço de uma tarefa. Dessa forma é possível, por exemplo, a utilizar ferramentas como as métricas de pontos de função, e garantir dados mais próximos da realidade, sem ter um grau de incerteza tão alto. Claro, as habilidades das pessoas mudam com o tempo, e também seu desempenho (seja para melhor ou pior), então é necessário ajustar estes valores sempre que um projeto terminar.

Caso não seja possível ter dados de muita qualidade, ainda sim é possível tentar:

  1. Aumentar as estimativas (Se obter 7 meses, peça 9, por exemplo)
  2. Técnicas de decomposição para cálculo
  3. Modelo empírico (experiências passadas)
  4. Ferramentas de estimativa

No caso das técnicas de decomposição, pode-se fazer um planejamento utilizando os pontos de função. Para isso é necessário decompor o problema de grande complexidade, em menores frações, então, com o auxílio da fórmula:

E = (a + 4b + c)/6, onde a = valor otimista, b = valor provável e c = valor pessimista.

Retorna um valor um pouco mais "embasado" do que apenas tentar chutar um valor e passar ao cliente.

Redes PERT-CPM

Esse é um modelo gráfico de representação de tarefas e eventos. Em outras palavras, com ele podemos descrever, por exemplo, o início de um projeto, seu fim e todo seu corpo, dizendo suas tarefas e o tempo que cada uma levou. Também representa a sequência lógica dessas tarefas, sendo elas lineares, dependentes ou paralelas.

A parte mais interessante desse modelo é o fato de nos trazer cálculos que mostram o tanto de tempo que podemos atrasar em determinadas tarefas, sem que alterem o cronograma final do projeto. Assim como também nos mostra, o caminho crítico, que nos diz onde no projeto temos as tarefas mais sensíveis, das quais, não se pode ter, quase nenhum ou nenhum, atraso, e que caso aconteça, implicam no atraso total do produto.

Temos alguns conceitos neste modelo, e são eles o cedo, o tarde, a folga do evento e o caminho crítico que foi dito anteriormente. O cedo é o tempo mínimo que o projeto levará para ser concluído e é calculado por max(cedo anterior + duração), do qual se inicia no primeiro evento e termina no último evento. O tarde é o tempo máximo que cada evento pode levar sem atrasar as datas de entrega do produto, é calculado por min(tarde posterior - duração). A folga é o tempo de sobra que cada evento tem antes de atrasar as entregas, e é calculado por tarde-cedo. E o caminho crítico que é o trajeto (início ao fim) com as menores folgas, das quais, a soma das durações de cada tarefa é igual ao cedo do último evento.

Mais alguns detalhes sobre o Projeto de Software - 05/05/2022

Autor: Rangel

Estrutura do Software

Existem muitos conceitos (muitos mesmo) no projeto de software, e eles servem exatamente para nos passar a experiência de outras pessoas que já tiveram diversos problemas quanto a gestão de um programa, aplicativo, etc. Neste caso, deixaram formas não apenas para os contornar, mas sim, evitar que eles aconteçam.

Existem diversos passos, procedimentos e estruturas criadas, que visam auxiliar na documentação e na codificação de um programa. Com isto, segue-se:

Abstração

Representar os procedimentos, dados e controle em vários níveis de detalhes. Se define uma solução usando linguagem de ambiente do problema.

Refinamento

melhoramentos sucessivos, trazendo níveis de detalhes cada vez melhores. Método top down (ver todos os detalhes genéricos e depois ir para os individuais) e bottom up (ver os detalhes individuais antes e depois ir abrangindo para algo geral).

Modularidade

Quebrar um problema maior em pequenos módulos gerenciáveis, afim de diminuir o grau de complexidade unitário. Estes tem diferentes níveis de independência funcional, e possuem como característica a coesão (se módulo faz apenas aquilo que foi desenvolvido para fazer) e o acoplamento (a função é chamada em outra quando necessária para fazer um cálculo, tarefa e etc (exemplo do foguete))

Arquitetura

Agrupa módulos que solucionam um problema do mundo real em algo mais genérico.

Hierarquia de Controle

Representa graficamente as chamadas de procedimentos dentro de um software (é como se mostrasse uma chamada que mostra outra, retorna para outra, etc, em formato de árvore)

Estrutura de dados

Representa vetores, matrizes, árvores, grafos e etc.

Procedimento de Software

Detalha o processamento de cada módulo individualmente, definindo os eventos, pontos de decisão e diversos outros conceitos relacionados a como cada procedimento/módulo age enquanto está exercendo sua função.

Ocultação de Informações

O que ficará visível das informações manipuladas em um método, exemplo do public, private e protected em orientação a objetos ou escopo em linguagens procedurais.

Métricas para definir custo e tempo de desenvolvimento

Este é um tópico amplamente abordado e que muitas pessoas não sabem nem por onde começar. Clientes adoram softwares baratos e que sejam rápidos para ficarem prontos, então é de responsabilidade da organização de parametrizar os requisítos e dizer o custo e o tempo que levará para ficar pronto. É muito importante fazer essa etapa, não apenas para satisfazer o mais exigente dos clientes, mas também para ter noção da viabilidade de um projeto, afinal, de nada adianta investir R$10.000 em um projeto que retornará R$2.000.

Existem diversas técnicas para definir esses valores, por exemplo, a utilização da contagem de linhas de código, onde, para cada certa quantidade de linhas, é considerado um tempo e um custo, entretanto, esse modelo serve única e exclusivamente para a linguagem da qual foi desenvolvido, então, para cada uma deve ser feito um estudo ao definir os valores. Felizmente, existem outros métodos e aqui estarei abordando o método de pontos de função.

Por mais que seja mais complexo, se torna abrangente e não depende de linguagem. A métrica orientada a função se baseia em pontuar por níveis de complexidade um software (simples, médio, complexo), levando em consideração aspectos mais técnicos como a quantidade dados, arquivos de erro, chaves, tabelas, telas e outros objetos, que ajudam a quantificar uma pontuação de custo e tempo estimados do software.

Aula sobre Projeto de Software - 28/04/2022

Autor: Rangel

Qual a importância do Projeto de Software?

Essa é uma etapa pré implementação. Nela são definidos os modelos que vão fazer parte do software final, definindo coisas como arquitetura, design e layout, algoritmos e outros. É importante fazer esse processo pois, além de poupar recursos financeiros, tempo e atribuir uma organização interessante aos projetos, evita que existam inconformidades e despadronizações no mesmo. Imagine que diversas equipes costumam trabalhar em paralelo em um mesmo projeto, sem essa abordagem, cada equipe ou até mesmo integrante, pode querer atribuir suas próprias interfaces, nomes de variáveis, tabelas em bancos de dados desconexas com os demais e diversos outros problemas que com certeza afetarão a qualidade final do produto.

Dramatização da importância do Projeto de Software. Disponível em: <https://docs.google.com/presentation/d/1_4DEr6vwFU6redoT8yYc2jrQDgFk-fWB/edit#slide=id.p6>. Acesso em: 30 abr. 2022.

Os tipos de projeto

Existem diversos tópicos a serem abordados quanto a criação de um projeto de software, e são divididos em duas categorias. Temos o ponto de vista gerencial (Projeto Preliminar, Projeto Detalhado) e o ponto de vista técnico, do qual, vou abordar alguns de seus tópicos nesta postagem.

  1. Projeto de Dados
  2. Projeto de Interface

Projeto de Dados

Nessa etapa a parte das informações do projeto são organizadas. Se verifica quais são os tipo de dados trabalhados assim como as estruturas também. Nesta parte, é feito um dicionário de dados, que serve como uma documentação especificando quais os tipos de variáveis usados no projeto, bem como suas funções, seu tipo, o que pode ou não receber e outros metadados atrelados a estes tipos. Existe também o dicionário para estrutura de dados, que especifica tipos que não são definidos por padrão, sendo assim, o desenvolvedor deve os criar. Um exemplo é um campo de um banco de dados que recebe itens como PDFs, GIFs, JPGs e outros, que não possuem uma implementação prévia por parte daqueles responsáveis pela criação de uma linguagem quaisquer.

Também é nesta etapa que são definidos os padrões a serem seguidos quanto a utilização de funções e variáveis dentro de um sistema. Como citado anteriormente, as chances de fracasso em um projeto são notavelmente maiores quando não existe uma padronização decente, pois cada um seguirá seus próprios padrões, o que trará incongruências, problemas de utilização dos recursos já criados, muitas vezes duplicidade de funções e diversos outros problemas.

Projeto de Interface

Este é um aspecto muito importante, ele que define muitas vezes se um projeto dará certo ou não, por parte do público. Uma coisa comprovada, é que a primeira coisa que as pessoas reparam ao utilizar um produto, é sua interface. Portanto, interfaces não atrativas afastam o público e consequentemente reduzem as chances de sucesso de um projeto.

Outro ponto é a facilidade e acessibilidade das funções em um software, de nada adianta uma interface bonita e muito confusa, da qual trás mais problemas do que soluções. Neste caso entram aspectos de design em UI e UX e que por muitas vezes são uma área completamente difundida na criação de produtos.

É importante cuidar muito bem dos aspectos gráficos em uma interface, as cores são um desses, já que muitas pesquisas demonstram que estas podem ser utilizadas para passar sensações, por exemplo, a cor preta passa um efeito de modernismo, enquanto a cor vermelha trás força, paixão, raiva e outros. Essa área de estudo se chama Psicologia das Cores.

Aula sobre Requisitos e Modelo de Casos de Uso - 21/04/2022

Autor: Rangel

Levantamento de requisitos, o que é?

Esse é um processo de suma importância feito sempre na primeira etapa de todo projeto, nele são debatidos junto a parte interessada, o que deve conter em determinada aplicação. Tais informações devem ser muito bem especificadas e são documentadas para serem utilizadas posteriormente. Dependendo do tipo de metodologia aplicada ao projeto, essa fase ocorre uma ou mais vezes até a entrega final do produto.

Para melhor compreender os requisitos, existem alguns tipos que são gerados enquanto nas fases de planejamento, e eles podem ser agrupados em:

  1. Funcionais
  2. Não funcionais
  3. Usuário
  4. Sistema

Requisitos de Usuário

São aqueles ditos pela parte interessada, da qual muitas vezes não tem conhecimento técnico, mas precisa de determinadas funções em um produto. Por exemplo, uma página que permite a criação e login de usuários, fazer postagens, comentários, mostrar notificações a aqueles que acessarem e outras funcionalidades. São genéricos e não especificam como deve ser feito, apenas o objetivo da funcionalidade.

Requisitos de Sistema

Já são feitos pelo time de desenvolvimento, que especificam de forma abstrata como determinada funcionalidade deve ser implementada, utilizando de determinadas ferramentas, linguagens e afins. Envolve a modelação de como o sistema deverá funcionar, com quais dados ele contará, os tipos dos dados e outras informações pertinentes para a implementação.

Requisitos Funcionais

São aqueles que alteram diretamente o funcionamento de um sistema e definem exatamente quais tarefas e funções que o mesmo terá, de forma específica e abstrata para cada item dos requisitos de sistema.

Requisitos Não Funcionais

São importantes, mas não implementam nada ao sistema diretamente, como por exemplo, verificações legais, os stackholders pedirem uma interface bonita e intuitiva, a utilização de determinados padrões impostos pelo cliente e diversos outros exemplos.

Exemplos de requisitos não funcionais. Disponível em: <https://docs.google.com/presentation/d/1Oxc47eupagmMn6kNpWrObO3RBi_aiucu/edit#slide=id.p4>. Acesso em: 29 abr. 2022.

Tipos de requisitos com FURPS+

Como já deve ter percebido, os requisitos tratam de diversas categorias diferentes e sem uma organização fica difícil de dizer do que se trata e qual sua contribuição ao projeto. Por conta disto, o modelo FURPS+ foi criado. Seu propósito é categorizar requisitos de modo a facilitar a compreensão dos mesmos e dizer qual utilidade terá dentro de um projeto.

Existem os seguintes tipos:

  1. Funcionalidade (Functionality)
  2. Usabilidade (Usability)
  3. Confiabilidade (Reliability)
  4. Desempenho (Performance)
  5. Suportabilidade (Supportability)

Funcionalidade

Característica definida, capacidades, a generalidade, a segurança.

Usabilidade

Fatores humanos, estética, coerência, documentação.

Confiabilidade

Gravidade de frequência de falha, de capacidade de recuperação, a previsibilidade, a rigor, tempo médio para falha.

Desempenho

Velocidade, eficiência, tempo de resposta de consumo, taxa de transferência de recursos.

Suportabilidade

Testabilidade, extensabilidade, adaptatibilidade, facilidade de manutenção, compatibilidade, configurabilidade, facilidade de manutenção, instalabilidade, localizabilidade, portabilidade.

O termo "+" no nome FURPS se refere a outros aspectos como implementação, interface, operações e questões legais.

Casos de uso

Essa é uma ferramenta, textual, muito interessante, com ela é possível definir atores e ações para completar determinadas tarefas em um sistema, como implementação de uma nova função. Por mais seja geralmente encontrado em versão gráfica, os autores dizem que este é um modelo textual de documentos, não diagramático, portanto vou estar tratando da versão em tabelas. Uma prática muito comum, chamada caso de uso caixa preta, é descrever de forma como o usuário descreveria uma determinada função. Por exemplo, ao dizer que um botão deve registrar um novo usuário, não se deve dizer como, nem o processo para isto acontecer, apenas, o que o botão faz, e neste caso, registrar uma nova conta. Existe também o caso de uso caixa branca, que é uma maneira completa de se descrever como um processo deve ocorrer para concluir aquela ação.

Exemplo de um caso de uso gráfico (por mais que a ferramenta seja textual). Disponível em: <https://pt.wikipedia.org/wiki/Diagrama_de_caso_de_uso>. Acesso em: 29 abr. 2022

A sua formalidade pode ser separada em:

  1. Resumido: um parágrafo (caso de sucesso)
  2. Informal: múltiplos parágrafos (vários cenários)
  3. Completo: descrição detalhada de condições e garantias de sucesso

Neste modelo existem os fluxos e desvios. Isso significa que a modelação já presupõe eventos que podem levar a resultados diferentes. Por exemplo, caso o usuário não digite um valor válido em uma caixa que apenas aceite dígitos.
